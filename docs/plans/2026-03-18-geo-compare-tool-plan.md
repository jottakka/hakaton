## What needs to be built

### Task 1 ŌĆö `geo_compare_output_schema.json` (new prompt asset)

**File:** `tools/benchmark_control_arcade/prompts/geo_compare_output_schema.json`

Extend the single-target schema to support a comparison run. Required top-level fields:

```json
{
  "target":    { "type": "string" },
  "competitors": { "type": "array", "items": { "type": "string" } },
  "run_date":  { "type": "string", "format": "date" },
  "audits": {
    "type": "array",
    "items": {
      "url":           "string",
      "overall_score": "integer  (0ŌĆō100)",
      "lever_scores": {
        "content_structure": "integer (0ŌĆō25)",
        "entity_authority":  "integer (0ŌĆō25)",
        "technical":         "integer (0ŌĆō25)",
        "citation":          "integer (0ŌĆō25)"
      },
      "artifacts": {
        "robots_txt":    "found|not_found|redirect|error",
        "sitemap_xml":   "same",
        "llms_txt":      "same",
        "llms_full_txt": "same"
      },
      "strengths":        ["string"],
      "weaknesses":       ["string"],
      "recommendations":  ["string"]
    }
  },
  "comparison_table": "string   (markdown table, generated by agent)",
  "winner_per_lever": {
    "content_structure": "string (url)",
    "entity_authority":  "string",
    "technical":         "string",
    "citation":          "string"
  },
  "overall_winner": "string (url)",
  "report_markdown": "string  (full human-readable comparative report)"
}
```

**Why:** The current `geo_site_audit_output_schema.json` has a single `target_url` field. A comparison run needs per-site scores in a structured array that the storage layer can index.

### Task 2 ŌĆö `geo_compare_system.md` (new prompt asset)

**File:** `tools/benchmark_control_arcade/prompts/geo_compare_system.md`

New system prompt for the competitive comparison agent. Must:
1. Call `CollectGeoEvidence` once per URL (target + all competitors) ŌĆö or in one batch call since `target_urls` already accepts a list.
2. Score each URL individually across all 4 levers.
3. Call `ValidateGeoAuditClaims` once per URL before scoring.
4. Produce a comparative markdown table showing all URLs side by side.
5. Identify the winner per lever and the overall leader.
6. Return a single JSON object conforming to `geo_compare_output_schema.json`.
7. Include `"Also runnable in Claude.ai web"` note ŌĆö the system prompt + tool calls must work with Claude's web interface when `geo-audit-arcade` MCP server is connected.

### Task 5 ŌĆö `SearchGeoReports` MCP tool

**File:** `tools/benchmark_control_arcade/src/benchmark_control_arcade/server.py` (add tool)  
**Helper:** `tools/benchmark_control_arcade/src/benchmark_control_arcade/history.py` (add function)

New MCP tool exposed on the `benchmark-control` server:

```python
@app.tool
async def SearchGeoReports(
    target: Annotated[str, "Site being audited (e.g. 'arcade.dev'). Empty = all."] = "",
    competitor: Annotated[str, "Filter runs that include this competitor URL. Empty = all."] = "",
    from_date: Annotated[str, "ISO-8601 start date inclusive (e.g. '2026-01-01'). Empty = no lower bound."] = "",
    to_date: Annotated[str, "ISO-8601 end date inclusive (e.g. '2026-12-31'). Empty = no upper bound."] = "",
    run_type: Annotated[str, "Filter by run type: 'geo', 'geo_compare', or empty for both."] = "",
    limit: Annotated[int, "Max results (default 20)."] = 20,
) -> Annotated[str, "JSON array of matching RunRecord summaries, newest first"]:
```

**Implementation:**
1. Call `client.list_run_records(limit=500)` to get all records.
2. Filter by `run_type in {"geo", "geo_compare"}`.
3. Filter by `created_at` date range if `from_date`/`to_date` provided.
4. Filter by `target` if provided (match against `record.spec.target`).
5. Filter by `competitor` if provided (match against `record.spec.options.get("competitors", [])`).
6. Return the top `limit` results as a JSON array of `RunRecord` dicts.

**Note:** No new GitHub API calls beyond what `list_run_records` already does. The path sort (`YYYY/MM/DD`) already gives us a fast date-range shortcut ŌĆö prune paths outside the date window before fetching.

## Architectural rules to preserve

- `server.py` must NOT have `from __future__ import annotations` ŌĆö Arcade `@app.tool` introspects annotations at decoration time.
- `Publisher` is the only code that writes files. `geo_compare_runner.py` calls `Publisher`, it does not build paths itself.
- `GitHubClient` is a boundary dependency. Test via `respx` at the HTTP boundary; do not patch internal helpers.
- No global manifest file. `SearchGeoReports` walks the git tree exactly like `ListRuns`.
- `geo_compare` runs are stored on the `benchmark-data` branch exactly like `geo` runs ŌĆö same layout, same `RunRecord` shape.
